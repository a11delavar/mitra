# AI Agent Instructions

When writing code for this project, you must adhere to the following architectural constraints:

- You are an expert in TypeScript, Lit, and modern CSS.
- Use standard Lit decorators (`@component`, `@property`, `@state`).
- Keep components small and reactive. Use Light DOM encapsulation (`createRenderRoot() { return this }`) to ensure seamless View Transitions and global themability.
- Do not use heavy frontend frameworks (React/Vue). Stick to standard web components.
- Write concise, performant code. Avoid over-engineering.
- For CSS, use standard nested CSS or modern native features (Container Queries, `color-mix`, native CSS variables).
- **Nested CSS:** All CSS rules MUST be nested to reflect the DOM hierarchy rather than using flat top-level class declarations.
- **Idiomatic HTML/CSS Naming:** Treat CSS class names as public APIs that could be externally themed. Avoid technical or structural terms like `btn`, `wrapper`, `container`, `layout`. Use concise, unified, and semantic idiomatic names that represent the domain (e.g., `.header`, `.entries`, `.time`, `.start`, `.end`, `.more`). Component templates should be as flat and simple as possible without redundant wrapper divs.
- When generating Calendar layouts, strictly use the CSS Grid approach.
- **Prefer CSS over JS:** If a layout can be solved with CSS Grid or Flexbox, do not use JavaScript math.
- **Calendar Grid:** The vertical time-grid uses `1440` rows. Map minutes directly to grid rows (e.g., 9:00 AM = row `540`).
- **Overlaps:** JS-based collision calculation is permitted *only* to assign `slot` and `total` overlaps. This data MUST be passed to the `<mitra-entry-segment>` instances strictly as CSS Custom Properties (e.g. `--overlap-slot`, `--overlap-total`), allowing CSS container queries to cleanly disable the clustering in narrow views (e.g., list views) without fighting JS logic.
- **Responsiveness:** Use CSS Container Queries (`@container`) instead of media queries. Components should adapt to their parent container's size, not the viewport.
- Strongly type all JSON API contracts from the backend proxies.
- **Dates and Times:** Always use the `@3mo/date-time` package and its `DateTime` global class (which utilizes the Temporal API) for any date, time, or formatting operations. Avoid raw `Date` objects or other libraries.
- **2D Scroll Architecture:** Use a single scrolling container (`overflow: auto`) for the calendar viewport. Achieve 2D locked headers via `position: sticky` (`top: 0` for day headers, `left: 0` for the time axis, and `top: 0; left: 0;` for the top-left corner). Do NOT sync separate X and Y scroll containers with JavaScript.
- **Dynamic Minute Sizing:** To avoid browser sub-pixel layout limits (like `1fr` flooring at 1px), enforce the vertical 1440-row grid to fit its container using dynamic percentages (`--minute-height: calc(100% / 1440)` on a `height: 100%` grid).
- **Entry Clustering (Interval Graphs):** When resolving parallel entries, use strict Interval Graph Coloring logic to segment clusters completely before calculating columns and spans. This prevents disjoint entries from falsely grouping via distant shared neighbors.
- **Cross-Day Entries:** Do NOT attempt to span a single DOM element across multiple days. Split entries crossing midnight at the data/selector layer (e.g., `getEntriesForDay`). Pass the *original unmodified time range* to the UI component for accurate text display, but use boolean flags (`continuesNext`, `continuedFromPrevious`) to mathematically clamp the grid rendering between rows `1` and `1441` and to strip border radii visually.
- **Layered domain/view separation:** `shared/` holds *persistence/domain records only* (`Entry`, `Source`, `Integration`/`CalDAV`) + API models ŌĆö no view concerns (do NOT put a calendar `laneRank` on `Entry`). An entity *may* expose facts about its own persistence, though ŌĆö e.g. `Entry.persisted` (has the backend assigned it an id yet) is domain state, not layout. Calendar layout is a **frontend** concern: `EntrySegment` (in `frontend/`, not `shared/`) is one entry projected onto a single day and owns its per-day geometry, all derived from `(entry, date, previous/next links)`. The `EntrySegments` cohort owns every *cross-segment* computation. Keep this math out of the rendering components ŌĆö `mitra-entry-segment`/`mitra-entry-details` are "decorations" that only read their segment for content/colour/identity while the view sets placement inline.
- **DateTime Operations:** Avoid raw `valueOf()` date math or manual year/month comparisons. Always use the native `@3mo/date-time` `DateTime` instance methods (`.equals()`, `.isBefore()`, `.isAfter()`, `.dayStart`) for bounding checks and equality. Rely on `DateTime.format()` to render strings so time values naturally inherit the system locale (e.g. Persian numerals).
- **RTL / Localization Support:** Always utilize CSS Logical Properties (e.g., `margin-inline-start`, `inset-inline-start`, `border-end-start-radius`, `padding-inline-end`) instead of physical directional properties (`margin-left`, `left`, `border-bottom-left-radius`) to ensure the application natively works in both LTR and RTL layouts. Avoid using `box-shadow` for asymmetrical borders (use pseudo-elements instead).
- **CSS Nesting / Shadow DOM Constraints:** Rather than fighting `:host([attr]) .child` selector specificity issues, attach boolean data attributes directly to structural markup (`<div ?data-state>`) and leverage native CSS nesting (`.child { &[data-state] { ... } }`).
- **Lit CSS Workaround:** Place all CSS directly within `static override get styles() { return css\`...\` }` rather than using `<style>` tags in the template. Use `import { css } from 'lit'` if `@a11d/lit` does not export it.
- **Scalable CSS Units:** Use `rem` instead of `px` values for most layouts (padding, margin, font-size, widths) to ensure responsive geometry. `px` should generally only be reserved for absolute rendering rules (like 1px borders or `@container` query thresholds).
- **CSS Variable Fallbacks:** Never hardcode default fallback values for theme variables (e.g., use `var(--color-background)` instead of `var(--color-background, #191919)`) since theme variables are guaranteed to exist globally.
- **Backend ORM Strategy:** Exclusively use SQLite with MikroORM. Utilize Single Table Inheritance (`@entity({ discriminatorColumn: 'type' })`) for polymorphic domain models like `Integration` and `Entry` to allow simple querying of complex inherited data. A new STI subclass that adds no columns (e.g. a new `Integration` discriminator) needs no migration ŌĆö just register it in `backend/orm.ts`'s `entities` and `shared/index.ts`.
- **Integration types:** `CalDAV` syncs against a remote server; `Dev` (`src/backend/Dev.ts`, `discriminatorValue: 'dev'`) is a backend-only, self-contained calendar with **no external backend** ŌĆö its `sync()` is a no-op (overridden so the empty `fetchSources` doesn't make the base `getSources` delete the locally-owned sources) and `createEntry`/`updateEntry`/`deleteEntry` are direct local persist/update/remove. It backs the dev sample data (`seedDev`) and lives in `backend/` (the frontend treats a `dev`-type integration as a generic one).
- **All-day:** `Entry.allDay` is a **persisted flag**, not inferred from midnight times. CalDAV reads it from a date-only `DTSTART` (`event.startDate.isDate`) on sync and writes `DTSTART;VALUE=DATE`/`DTEND;VALUE=DATE` (exclusive next day) on create/update; `start`/`end` still carry the day bounds (midnight ŌåÆ exclusive next midnight) for rendering. Always set the flag where an entry is created (drag, seed, sync).
- **Background Sync Architecture:** All data synchronization should occur entirely in a background daemon (e.g., `server.ts` loop) that polymorphically iterates over all base `Integration` records. Do NOT block frontend API reads with live external syncs; the frontend `/api` routes must always query instantly from the local SQLite store.
- **CalDAV Sync Quirks:** When syncing CalDAV via `tsdav`, **do not** use `smartCollectionSync` or `smartCollectionSyncDetailed`. It has a hardcoded bug that filters out any calendar URLs that do not end in `.ics`, breaking platforms like some CalDAV servers. Always implement manual WebDAV XML syncing by passing `client.syncCollection()` directly and fetching `changedUrls` manually.
- **Opt-in Data Flow:** Newly discovered external sources (like new calendars) should always be persisted to the DB with `enabled: false`. Do not perform heavy `Entry` syncing loops until the user explicitly enables a source.
- **Living Architecture:** Whenever a new architectural decision is made by the user, **ALWAYS** add it to this `AGENTS.md` file immediately so that rules compound and the user doesn't have to repeat themselves.

## Multi-user & authentication (OIDC)

- **Zero-auth single-user is the default**: `authMiddleware` attaches the seeded `[default_local_user]` to every request. Setting `MITRA_OIDC_ISSUER` (+ `MITRA_OIDC_CLIENT_ID`, `MITRA_URL`, optional `MITRA_OIDC_CLIENT_SECRET`/`MITRA_OIDC_SCOPES`) switches on multi-user mode; a half-configured issuer fails the boot loudly (`Oidc.fromEnv`).
- **The BACKEND is the OIDC relying party** (`backend/Oidc.ts`, Authorization Code + PKCE via `openid-client`, routes under `/auth/*`) ŌĆö deliberately NOT frontend `oidc-client-ts` + Bearer tokens: the SPA is same-origin (one Express serves both), `@a11d/api` already sends cookies, and tokens staying server-side beats XSS-exposed browser storage. Discovery is lazy and retried (the IdP may boot after mitra in the same compose stack); an `http:` issuer is allowed for LAN/compose-internal IdPs.
- **Sessions are mitra's own** (`backend/Session.ts`): an opaque 256-bit cookie token stored SHA-256-hashed as the row id, sliding 30-day expiry. The IdP's tokens are discarded after sign-in ŌĆö mitra calls no upstream APIs for the user ŌĆö except the raw `id_token`, kept per session as the `id_token_hint` for RP-initiated logout. CSRF rests on `SameSite=Lax`. Unauthenticated requests split by kind: `/api/*` ŌåÆ 401 (the frontend's `ApiError` bounces through `/auth/login`), asset paths (with an extension) pass ŌĆö bundles and the PWA manifest are public code ŌĆö and page navigations redirect to `/auth/login?returnTo=ŌĆ”`.
- **The OIDC identity is a value object, not flat columns**: `Identity` (`shared/Identity.ts`) holds `issuer`/`subject` (the key ŌĆö a subject is unique only within its issuer) plus the vouched `email`/`name`/`picture` (the photo is stored as a LINK only ŌĆö the browser loads it from the provider), embedded into `User` as `oidc_*` columns via `@embedded(() => Identity, { prefix: 'oidc_', nullable: true })` ŌĆö the same MikroORM embeddable pattern as `Recurrence` on `Entry` (registered in `orm.ts` entities; unique on `['identity.issuer', 'identity.subject']`). Absent entirely in single-user mode, so `user.identity` presence IS the "is this an OIDC user" test (frontend + backend). Behavior lives on it: `Identity.fromClaims` / `applyClaims`.
- **JIT provisioning is domain behavior**: `User.provision(em, issuer, claims)` matches on the embedded `(issuer, subject)` and creates a fresh user on first sight. `email`/`name` refresh from the ID token on every sign-in. **No data migration**: enabling OIDC does NOT carry the single-user default user's data over ŌĆö every identity (incl. the first) starts empty and re-adds its own integrations. This is a deliberate call (the user's): don't write upgrade/backfill code; tell users to reimport.
- **Ownership scoping lives on `User`**: routes resolve entities exclusively through `user.integrations/integration/sources/source/entry` ŌĆö a foreign id throws `NotFoundError` ŌåÆ the central 404, never leaking another user's data. Never use a bare `em.find*` for user-owned entities in a route unless ownership was already proven upstream in that handler.
- **Everything user-adjacent is per-user**: `syncEmitter.emit('updated', userId)` (the SSE feed filters by `req.user.id`; `Synchronizer` collects changed owners per integration), push subscriptions carry a required `userId` set at subscribe time (re-registration on boot reassigns a shared browser to whoever is signed in) and reminders route via entry ŌåÆ source ŌåÆ integration ŌåÆ owner (`sendTo(userId, ŌĆ”)`).

## Calendar layout & segmentation (current model)

- **View components are named after the unit they strip**: `Days` (`mitra-days`) is the week view's horizontal strip of day columns; `Weeks` (`mitra-weeks`) is the month view's vertical strip of week rows. A future view follows the same scheme (e.g. a year view = a strip of months). The user-facing *view* names ("Week"/"Month") and the drag controller's `grid` discriminants stay as-is.
- **`EntrySegments` (frontend cohort) is the whole layout engine** ŌĆö small, because CSS does the placing. Build via `EntrySegments.of(entries, days)` (memoised on `days`). API:
  - `EntrySegments.for(entry)` ŌĆö the entry's per-day segments, linked `previous`/`next`, memoised so instances are stable across renders.
  - `timedOn(day)` ŌĆö the day's timed segments clustered into side-by-side columns (the *one* genuinely cross-segment computation; writes each segment's `overlap = {slot,total,span}`).
  - `runsIn(from, to, accept)` ŌĆö one representative segment per matching entry whose run touches the window, sorted so DOM order drives CSS `grid-auto-flow: dense` lane packing. Used for the week all-day lane and per-week in month.
  - `monthSlots` / `monthWeek(week, maxSlots)` ŌĆö month packing. Month keeps a *computed* slot (unlike the all-day lane's pure CSS auto-flow) because its "+N more" overflow needs exact lanes.
  - `static laneRank(entry)` ŌĆö lane ordering for month packing (a view concern, hence here and not on `Entry`).
- **Grid gestures ŌĆö one `EntryDragController`, both views, three gestures.** Attached to the grid *container* (`Days`/`Weeks`), never per-`Day`, so a gesture can span day cells; the view selects the surface with a `grid: 'week' | 'month'` arg. It snapshots cell geometry once at pointer-down (so each move is pure arithmetic ŌĆö no per-frame reflow), coalesces moves to one update per animation frame, captures the pointer on the host, and tears down on pointerup **or pointercancel**. Pointer-down classifies the gesture once (a `kind` discriminant): a `.resize-start`/`.resize-end` handle ŌåÆ **resize**, a persisted entry's body ŌåÆ **move**, empty grid/lane/cell ŌåÆ **create**. It also **owns tap-to-open** (a no-move tap on a body opens the editor) ŌĆö `mitra-entry-segment` has no click handler, so a drag can't fire a stray open on the wrong slice. Geometry/granularity are shared: timed only in the week's grid (minutes off the 1440-row grid, snapped to `snapMinutes`); the all-day lane and the *whole* month are day-granular (a timed entry moved in the month shifts whole days, keeping its clock time).
  - **create** ŌĆö week timed/all-day lane/month, as before (plain click: nothing in week, quick-create a day in month). New entries target the dev sample calendar in development.
  - **move** ŌĆö translate a persisted entry preserving duration (clone both ends by one delta; the result's leading edge snaps to grid). Allowed for every entry, including timed-in-month.
  - **resize** ŌĆö drag one edge while the other stays fixed, reusing the create min/max-flip placement (`entryPlacement.ts`), so dragging an edge past the other flips gracefully and multi-day spans fall out for free. **Pure placement math lives in `entryPlacement.ts`** (`placeTimed`/`placeAllDay`/`resizePlacement`/`snapToGrid`) ŌĆö dependency-free and unit-tested (the all-day exclusive-next-midnight bound makes the fixed *trailing* edge `end ŌłÆ 1 day`).
  - **Resize handles** are 0.25rem strips rendered by `mitra-entry-segment` per a `resize: 'block' | 'inline'` prop the view sets (Day timed ŌåÆ block/vertical, all-day lane + month all-day bars ŌåÆ inline/horizontal, **month timed bars ŌåÆ none**, gated on the entry's *persisted* `allDay` flag, not the segment's derived one). CSS hides a handle at a clipped/interior edge via the same `has-previous`/`has-next` attributes (so `MonthBar` exposes `clippedRight`) and drops block handles on too-short entries.
- **Drafts & optimism ŌĆö one pending local entry, new *or* edited.** A locally-**created** entry (drag/click) has **no id** (the backend assigns one on create, so `Entry.persisted` = `id != null` *is* the draft test ŌĆö no flag, no status enum). An **edit** (move/resize) is a *clone* of an existing entry with mutated start/end (same id) ŌĆö never an in-place mutation, so a cancelled gesture leaves the original untouched. `DraftController` (static store) holds the one pending entry: `mergeInto` substitutes it over the server copy by id (an edit) or appends it (a not-yet-saved create); `reconcile` (called from the page's fetch) drops it once the server echoes the **matching content** (start/end/allDay) ŌĆö id alone would drop an edit's overlay before its update lands and snap it back. **Commit:** create ŌåÆ `createEvent` (no id), set the returned id in place; move/resize ŌåÆ `updateEvent(clone)`, reverting *that* draft on failure. A `dragging` flag floats the actively-dragged entry above its cluster. Components attach an instance only to **subscribe**.
- **Backend owns ids:** `Entry.id` has no default; assign one at every persist site (the POST route ŌĆö which covers both providers' `createEntry` ŌĆö plus `seedDev` and CalDAV sync). The frontend never mints ids; a missing id means "not yet saved".
- **Hot-loop date math:** `@3mo/date-time` rebuilds a `Temporal.Instant` **and** `ZonedDateTime` on *every* accessor (`.hour`/`.dayStart`/`.equals`/`.isBefore`ŌĆ”), which dominates per-frame render/clustering loops. So compare **cached epoch-ms day numbers** instead: `EntrySegment` memoises its numeric projections (`startMinute`/`endMinute`/`dayValue`) and segments are globally memoised (`EntrySegments.for`), so the Temporal cost is paid once per segment; compare `dayValue`/`fallsOn(dayValue)` (or hoist a `dayStart.valueOf()` out of the loop) rather than `dayStart.equals(...)`. The richer `DateTime` comparison API still applies to non-hot code.
- **Self-placement:** views map a date ŌåÆ grid column themselves (build a `Map<dayValue, index>` once per render ŌĆö never `findIndex` inside `.map()`) and set `grid-column`/`grid-row`/`--overlap-*` inline. The all-day lane self-places columns by date and lets `grid-auto-flow: row dense` assign lanes (no JS lanes). `CalendarLayout` (an old shared class) was removed ŌĆö don't reintroduce it.

## Command palette

- **`mitra-command-palette` is a pure view; commands belong to their context.** The `Command` interface (`frontend/Command.ts`: heading, icon, optional shortcut hint + match keywords, `execute`) is defined by the page that owns the behavior ŌĆö `PageCalendar` builds its command list per render (so view-dependent labels like "Next Week/Month" stay current) and passes it in as a property. The palette only filters (`commandMatches`: every query term must appear in heading+keywords), navigates (arrows/Enter, hover) and dispatches. Picking an entry emits `navigate` with its start **and** calls `EntryStore.requestOpen(id)` so its editor pops open once its segment renders; picking a command runs `execute()` after closing.
- **Opening a picked entry reuses the draft auto-open mechanism, generalized.** `EntryStore.requestOpen(id)` sets a pending `openEntryId` (sibling to the draft's `autoOpen` flag); `EventSegment.updated` opens the run-start segment (`!hasPrevious`, so a multi-day entry opens once) when `shouldOpen(entry)` matches, then `consumeOpen()`s it. The id is kept until consumed so it survives the async refetch the navigation triggers (cross-month picks work), and `shouldOpen` matches `entry.recurrenceMasterId === openEntryId` too, so picking a recurring master opens whichever occurrence the navigation lands on.
- **Entry search is backend-side and unwindowed** (`GET /entries/search?q=`): LIKE over heading/description/location across visible sources (limit 20, newest first), because the calendar only ever fetches ┬▒1 month ŌĆö the palette must find *everything*. Recurring masters match as themselves (one row stands in for its series). Debounced 200ms with a monotonic token so only the latest response lands.
- **Open affordances:** Ctrl/Cmd+P (window keydown listener owned by the palette itself; `CommandPalette.hotkey` renders the platform label) and the header's fake search box ŌĆö a `button.search` styled like an input, centered between flex spacers, collapsing to an icon-only button under `@container (max-width: 40rem)` (the header is an inline-size container).
- **It's a native `<dialog>` with `closedby="any"`** for light dismiss ŌĆö no hand-rolled backdrop-click handler (that pattern closes on the *trigger's* double-click). Styling follows the popover look: translucent `color-mix` surface + `backdrop-filter: blur`, `@starting-style` entry animation.
- **"Create Entry" reuses the gesture-draft flow** (`EntryStore.upsertDraft` + `openDraft`): a blank one-hour draft at the next full hour today on `getPrimarySource()` (Api.ts ŌĆö default-or-first-visible; also what the drag controller uses). It always `setView('week')`s first: the week's timed grid renders every draft, whereas a crowded month cell folds it into "+N more" and the auto-open editor would never appear.

## Recurrence (RFC 5545)

- **One master row, occurrences expanded on read ŌĆö never stored.** A recurring series is a single `Entry` carrying `rrule` (+ `uid`); `CalDAV.expandRecurrence(raw, windowStart, windowEnd)` iterates the rule into the occurrences intersecting the visible window. `GET /entries` **excludes masters** (`rrule != null`) from the plain rows and emits synthetic per-occurrence `Entry` objects instead, so the bare master (whose `DTSTART` is only its first instance) is never rendered on its own. Occurrences are non-persisted, with a **CSS-ident-safe id** `` `${master.id}__${epochMs}` `` (underscores + digits ŌĆö colons/dots break `anchor-name`/`view-transition-name`) and `recurrenceMasterId` set.
- **Overrides & exclusions.** A single edited occurrence syncs from CalDAV as its own row with a `RECURRENCE-ID`, linked to its master by shared `uid` (a post-loop pass in `syncSourceEntries`, scoped to the source); the GET skips expanded dates that already have an override row. `EXDATE`s are applied during expansion (matched by epoch-ms).
- **Editing is series-wide (current version).** There is no per-occurrence editing yet. Editing a rendered occurrence routes to the master via `recurrenceMasterId` (`Api.updateEvent` sends only series-wide fields, so the master keeps its own `start`/`end`); deleting removes the whole series. **`Entry.partOfSeries`** (`!!rrule || isRecurring`) is the intrinsic test for "belongs to a series" ŌĆö its schedule (date/time/all-day) is read-only and it isn't drag/resize-movable; only the Repeat field stays editable. Setting a repeat on a non-recurring entry makes it a master (its own id; the schedule then locks ŌĆö remove the repeat to edit times again).
- **RRULE strings are authored on the frontend.** `src/frontend/recurrence.ts` owns the whole RRULE subset: `buildRRule`/`parseRRule`, the date-derived quick presets (`recurrencePresets`), structural equality/match, and the human `recurrenceLabel`. `mitra-repeat-field` (the Notion-style trigger + preset dropdown + Custom dialog) mutates `entry.rrule` in place and fires a distinct `recurrenceChange`. The **backend never builds rules** ŌĆö it just writes the string onto the master's `.ics`: `CalDAV.createEntry`/`updateEntry` set it via `ICAL.Recur.fromString(rrule)` and drop it (collapsing the series) via `removeAllProperties('rrule'|'exdate')`. The routes validate it first (`CalDAV.isValidRRule`, 400 on junk) so a bad rule can't 500 the `.ics` writer.
- **Recurrence edits go through a dedicated path, never the content-edit path.** `Api.updateEvent` (heading/description/colour/status, plus the move-to-master routing for occurrences) **never writes `rrule`**; only `Api.setRecurrence` does (string = set, `null` = remove; `entries.ts` PUT maps absent = keep). This split is load-bearing: a synced *override* (an externally edited single occurrence ŌĆö `recurrenceMasterId` set, but no `rrule` of its own) renders as an editable row, and inferring `rrule` from it on a benign rename would otherwise wipe the master's whole series. So content edits must stay rule-silent.
- **Expansion works without an `.ics` too.** `expandRecurrence(raw,ŌĆ”)` (CalDAV, carries EXDATEs/anchor from the stored `.ics`) and `expandRecurrenceFields(rrule, start, end, ŌĆ”)` (column-only, for integrations like `Dev` that persist no `.ics`) share one `iterateOccurrences` core; `entries.ts` picks the column path when `master.data.raw` is absent. **ical.js 2.x has no `Recur.between()`** ŌĆö the core iterates `rrule.iterator(anchor)` and stops once occurrences ascend past the window, with a generous iteration cap (a far-future window must still be *reachable* for a dense daily series) and a no-advance guard against a malformed rule.
- **Backfill:** entries synced before recurrence support have the `RRULE`/`RECURRENCE-ID` in `data.raw` but NULL columns, and the etag-skip won't re-read them ŌĆö `backend/orm.ts` parses the candidates once on boot and links overrides to masters by UID (idempotent).

## Build / test / run

- **Node 25+** is required at build & runtime (Temporal API behaviour) ŌĆö pinned in the Dockerfile and CI.
- **Typecheck (one-shot):** run `tsgo` (`node_modules/@typescript/native-preview-<platform>/lib/tsgo[.exe] --noEmit`). esbuild does NOT typecheck.
- **Lint:** `npm run lint` ŌåÆ `eslint .` (flat config `eslint.config.mjs` extending `@a11d/eslint-config`). Pinned to ESLint 9 (`eslint-plugin-lit@1` breaks on ESLint 10). Style is enforced: tabs, single quotes, **no semicolons**, **no trailing newline** (`eol-last: never`), `no-console` allows only `warn`/`error` (off in `scripts/**`), `^_`-prefixed args are ignored. Run `eslint . --fix` after edits.
- **Tests:** `npm test` ŌåÆ `scripts/test.ts` esbuild-bundles `src/**/*.test.ts` ŌåÆ `out_test/`, then `node --test`. Use the Node built-in runner (`node:test` + `node:assert/strict`) with real `DateTime`.
- **Dev:** `npm start` ŌåÆ `scripts/dev.ts`: `tsgo --watch` + esbuild watch for backend (`out/server/server.mjs`, run via `node --watch`) and frontend (`dist/`).
- **Production build:** `npm run build` ŌåÆ `scripts/build.ts` (one-shot, no watch): backend bundle + minified frontend ŌåÆ `dist/`. The esbuild config (entry points, externals `better-sqlite3`/`tsdav`, the `createRequire` banner, the `index.html` shell) is shared between dev and prod in `scripts/esbuild.ts` ŌĆö change it there, not in two places. `data/` must exist before boot (better-sqlite3 won't create it); a committed `data/.gitignore` keeps the dir, and the Dockerfile `mkdir`s it.
- **Dev sample data:** `src/backend/Dev.ts` `seedDev(orm)` **persists** a real `Dev` sample integration + sources + entries (idempotent, keyed by a stable id; never deletes), gated by `process.env.MITRA_DEV` (set only by `scripts/dev.ts`). It seeds real rows ŌĆö *not* a read-only overlay ŌĆö so hiding sources, editing, creating and deleting all work through the normal routes. The user can simply delete the integration when done. Seeding runs once; a `npm start` restart re-checks but won't duplicate.

- **Tagging is `docker/metadata-action`, not hand-rolled** (`.github/workflows/docker.yml`): push to `main` ŌåÆ `dev` + `sha-<short>`; **push tag `vX.Y.Z`** ŌåÆ `X.Y.Z`/`X.Y`/`X`/`latest`; pre-release tag `vX.Y.Z-rc.N` (channels `rc`/`beta`/`alpha`) ŌåÆ only the precise tag (`flavor: latest=auto` auto-skips floating tags + `latest` for pre-releases); PRs build-only. Multi-arch amd64/arm64 via QEMU+buildx. Image tags come off the **git ref**, so they don't depend on a GitHub Release existing.
- **Changelog lives in the repo** as `CHANGELOG.md`, generated from commits by **git-cliff** (`npm run changelog`, config `cliff.toml`) and then hand/AI-editable. Release flow: `npm run changelog -- --tag vX.Y.Z` ŌåÆ edit the new section ŌåÆ `commit -m "release: vX.Y.Z"` ŌåÆ `git tag vX.Y.Z && git push origin main vX.Y.Z`. **Order matters:** the changelog edit must be committed *before* the tag points at it (the release body is extracted from the file at the tagged commit).
- **Releasing = pushing the SemVer tag.** That one push fans out (in parallel) to `docker.yml` (builds images off the git ref) and `release.yml` (publishes the top section of `CHANGELOG.md` as the GitHub Release body; pre-release if the tag has `-rc`/`-beta`/`-alpha`). Tag-driven on purpose ŌĆö avoids the `GITHUB_TOKEN`-created-release-doesn't-trigger-workflows gotcha. The README "Releasing & versioning" section is the user-facing source of truth.
- **GHCR has no GC** ŌåÆ `.github/workflows/cleanup.yml` prunes untagged manifests + old `sha-*` tags (keeps `latest`/`dev`/releases). Ships in `dry-run` ŌĆö verify before enabling.
- **Quality Assurance** (`.github/workflows/qa.yml`): three **parallel** jobs ŌĆö type-check (`tsgo`), lint (`eslint`), test (`npm test`) ŌĆö on push/PR. Publishing untested images is not acceptable.
