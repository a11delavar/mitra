# AI Agent Instructions

When writing code for this project, you must adhere to the following architectural constraints:

- You are an expert in TypeScript, Lit, and modern CSS.
- Use standard Lit decorators (`@component`, `@property`, `@state`).
- Keep components small and reactive. Use Light DOM encapsulation (`createRenderRoot() { return this }`) to ensure seamless View Transitions and global themability.
- Do not use heavy frontend frameworks (React/Vue). Stick to standard web components.
- Write concise, performant code. Avoid over-engineering.
- For CSS, use standard nested CSS or modern native features (Container Queries, `color-mix`, native CSS variables).
- **Nested CSS:** All CSS rules MUST be nested to reflect the DOM hierarchy rather than using flat top-level class declarations.
- **Idiomatic HTML/CSS Naming:** Treat CSS class names as public APIs that could be externally themed. Avoid technical or structural terms like `btn`, `wrapper`, `container`, `layout`. Use concise, unified, and semantic idiomatic names that represent the domain (e.g., `.header`, `.entries`, `.time`, `.start`, `.end`, `.more`). Component templates should be as flat and simple as possible without redundant wrapper divs.
- When generating Calendar layouts, strictly use the CSS Grid approach.
- **Prefer CSS over JS:** If a layout can be solved with CSS Grid or Flexbox, do not use JavaScript math.
- **Calendar Grid:** The vertical time-grid uses `1440` rows. Map minutes directly to grid rows (e.g., 9:00 AM = row `540`).
- **Overlaps:** JS-based collision calculation is permitted *only* to assign `slot` and `total` overlaps. This data MUST be passed to the `<mitra-entry-segment>` instances strictly as CSS Custom Properties (e.g. `--overlap-slot`, `--overlap-total`), allowing CSS container queries to cleanly disable the clustering in narrow views (e.g., list views) without fighting JS logic.
- **Responsiveness:** Use CSS Container Queries (`@container`) instead of media queries. Components should adapt to their parent container's size, not the viewport.
- Strongly type all JSON API contracts from the backend proxies.
- **Dates and Times — two vocabularies, one boundary:** the FRONTEND (rendering, formatting, view math) uses `@3mo/date-time`'s `DateTime` global; SHARED/BACKEND domain code uses **Temporal types directly** (`Temporal.Instant` for absolute times, `Temporal.PlainDate` for calendar dates, `Temporal.ZonedDateTime` for wall-clock math — the full polyfill is injected into every bundle by `scripts/injectTemporalPolyfill.ts`, so shared code may use it freely). Never hand-roll zone/day arithmetic (Intl `formatToParts` loops, `± 86_400_000`, noon-UTC formatting tricks) — `shared/calendarDate.ts` holds the two date↔instant primitives, `calendarDateOf(instant, zone) → Temporal.PlainDate` and `midnightOf(plainDate, zone) → Date`, plus the named all-day API-boundary pair `normalizeAllDay`/`projectAllDay` built on them; date math beyond that is `PlainDate` instance methods (`since`, `subtract`, `equals`, `toLocaleString`). **Never spread a `PlainDate`** — its fields are prototype getters and spread to `{}`. `ICAL.Time` stays strictly at the .ics parse/serialize boundary (converted immediately via `CalDAV.instantFrom`/`toICALTime`); zoned .ics WRITES keep the resource's own VTIMEZONE (TZIDs aren't guaranteed IANA — Outlook writes `W. Europe Standard Time`). Avoid raw `Date` beyond storage/interop signatures.
- **2D Scroll Architecture:** Use a single scrolling container (`overflow: auto`) for the calendar viewport. Achieve 2D locked headers via `position: sticky` (`top: 0` for day headers, `left: 0` for the time axis, and `top: 0; left: 0;` for the top-left corner). Do NOT sync separate X and Y scroll containers with JavaScript.
- **Dynamic Minute Sizing:** To avoid browser sub-pixel layout limits (like `1fr` flooring at 1px), enforce the vertical 1440-row grid to fit its container using dynamic percentages (`--minute-height: calc(100% / 1440)` on a `height: 100%` grid).
- **Entry Clustering (Interval Graphs):** When resolving parallel entries, use strict Interval Graph Coloring logic to segment clusters completely before calculating columns and spans. This prevents disjoint entries from falsely grouping via distant shared neighbors.
- **Cross-Day Entries:** Do NOT attempt to span a single DOM element across multiple days. Split entries crossing midnight at the data/selector layer (e.g., `getEntriesForDay`). Pass the *original unmodified time range* to the UI component for accurate text display, but use boolean flags (`continuesNext`, `continuedFromPrevious`) to mathematically clamp the grid rendering between rows `1` and `1441` and to strip border radii visually.
- **Layered domain/view separation:** `shared/` holds *persistence/domain records only* (`Entry`, `Source`, `Integration`/`CalDAV`) + API models — no view concerns (do NOT put a calendar `laneRank` on `Entry`). An entity *may* expose facts about its own persistence, though — e.g. `Entry.persisted` (has the backend assigned it an id yet) is domain state, not layout. Calendar layout is a **frontend** concern: `EntrySegment` (in `frontend/`, not `shared/`) is one entry projected onto a single day and owns its per-day geometry, all derived from `(entry, date, previous/next links)`. The `EntrySegments` cohort owns every *cross-segment* computation. Keep this math out of the rendering components — `mitra-entry-segment`/`mitra-entry-details` are "decorations" that only read their segment for content/colour/identity while the view sets placement inline.
- **DateTime Operations:** Avoid raw `valueOf()` date math or manual year/month comparisons. Always use the native `@3mo/date-time` `DateTime` instance methods (`.equals()`, `.isBefore()`, `.isAfter()`, `.dayStart`) for bounding checks and equality. Rely on `DateTime.format()` to render strings so time values naturally inherit the system locale (e.g. Persian numerals).
- **RTL / Localization Support:** Always utilize CSS Logical Properties (e.g., `margin-inline-start`, `inset-inline-start`, `border-end-start-radius`, `padding-inline-end`) instead of physical directional properties (`margin-left`, `left`, `border-bottom-left-radius`) to ensure the application natively works in both LTR and RTL layouts. Avoid using `box-shadow` for asymmetrical borders (use pseudo-elements instead).
- **CSS Nesting / Shadow DOM Constraints:** Rather than fighting `:host([attr]) .child` selector specificity issues, attach boolean data attributes directly to structural markup (`<div ?data-state>`) and leverage native CSS nesting (`.child { &[data-state] { ... } }`).
- **Lit CSS Workaround:** Place all CSS directly within `static override get styles() { return css\`...\` }` rather than using `<style>` tags in the template. Use `import { css } from 'lit'` if `@a11d/lit` does not export it.
- **Scalable CSS Units:** Use `rem` instead of `px` values for most layouts (padding, margin, font-size, widths) to ensure responsive geometry. `px` should generally only be reserved for absolute rendering rules (like 1px borders or `@container` query thresholds).
- **CSS Variable Fallbacks:** Never hardcode default fallback values for theme variables (e.g., use `var(--color-background)` instead of `var(--color-background, #191919)`) since theme variables are guaranteed to exist globally.
- **Backend ORM Strategy:** Exclusively use SQLite with MikroORM. Utilize Single Table Inheritance (`@entity({ discriminatorColumn: 'type' })`) for polymorphic domain models like `Integration` and `Entry` to allow simple querying of complex inherited data. A new STI subclass that adds no columns (e.g. a new `Integration` discriminator) needs no migration — just register it in `backend/orm.ts`'s `entities` and `shared/index.ts`.
- **Integration types:** `CalDAV` syncs against a remote server; `GoogleCalendar` (`shared/GoogleCalendar.ts`, `discriminatorValue: 'google'`) **extends `CalDAV`** — the whole sync/CRUD engine is inherited and only authentication differs (see below); `Dev` (`src/backend/Dev.ts`, `discriminatorValue: 'dev'`) is a backend-only, self-contained calendar with **no external backend** — its `sync()` is a no-op (overridden so the empty `fetchSources` doesn't make the base `getSources` delete the locally-owned sources) and `createEntry`/`updateEntry`/`deleteEntry` are direct local persist/update/remove. It backs the dev sample data (`seedDev`) and lives in `backend/` (the frontend treats a `dev`-type integration as a generic one).
- **Google Calendar = CalDAV + OAuth, not a new engine.** Google's CalDAV v2 endpoint (`https://apidata.googleusercontent.com/caldav/v2/`) supports everything mitra uses (VEVENT CRUD, `sync-collection`); it just refuses Basic auth. So `GoogleCalendar` overrides only the `protected clientParameters()` hook (the tsdav config `CalDAV.getClient()` builds from) to `authMethod: 'Oauth'` — tsdav mints/refreshes Bearer tokens itself from a stored `refreshToken` + the deployment's `MITRA_GOOGLE_CLIENT_ID`/`MITRA_GOOGLE_CLIENT_SECRET` (env, read lazily server-side; the shared class is also the frontend API model). Credentials are `{ username: email, refreshToken }` — the email doubles as the sidebar label and derives the per-account `uri` (`GoogleCalendar.uriFor`), which the `(userId, uri)` unique constraint uses to make reconnecting the same account an in-place grant renewal. **The refresh token never leaves the server**: `toJSON` masks credentials down to the username, and `merge` is a no-op (nothing is form-editable; a reconnect re-runs consent).
- **The Google consent flow is the backend's** (`backend/GoogleOAuth.ts` + routes in `integrations.ts`): `GET /api/integrations/google/connect` → Google (`access_type=offline&prompt=consent`, PKCE + state in a short-lived transit cookie, same pattern as `auth.ts`) → `GET /api/integrations/google/callback` exchanges the code (via `openid-client` against `accounts.google.com`), upserts the integration, pre-discovers its calendars (persisted disabled, per the opt-in flow), and redirects to `/?integration=<id>` — `Mitra.initialized()` consumes that param (BEFORE the router's first render, which would otherwise re-push it) and opens the source-picker dialog with every source pre-ticked. `GET /api/integrations/google` reports `{ configured }` so the add dialog only offers a working provider. It's deliberately separate from sign-in OIDC: a per-user data grant, present in single-user mode too.
- **Notion = database VIEWS as task sources, token auth, own engine** (`shared/Notion.ts` + `shared/NotionClient.ts`, `discriminatorValue: 'notion'`): a direct `Integration` subclass (no CalDAV — Notion is a JSON REST API, pinned `Notion-Version: 2026-03-11`). Auth is a pasted internal-connection/PAT token (zero deployment config; the `(userId, uri)` identity is the token's bot user id, resolved as a `fetchSources` side effect via `GET /users/me`, which also yields the workspace label as `credentials.username`; `toJSON` masks the token like Google's grant). Sources are `notion://{dataSourceId}/{viewId}` — only databases with a **status AND a date** property qualify; Notion evaluates view filters server-side, so a source's membership is `POST /views/{id}/queries` (partial page refs). `NotionClient.views()` lists a data source's views (partial refs, no type) then GETs each for its name/type — and a view whose type the API refuses to serialize (`400 validation_error: Unsupported view type: feed`, and any future kind) is DROPPED, not fatal: one unsupported view must not abort discovery/sync for the whole account (a real "cannot connect Notion at all" regression when a user adds a feed view). Sync is **full-membership + incremental content**: deletions are the membership set difference (guarded: skipped when the membership is truncated at the 10k cap, and rows edited <2min ago are never removed — view indexing lag would otherwise eat just-created pages), content comes from ONE data-source delta query (`last_edited_time on_or_after` watermark−2min overlap, ascending sort so a truncation keeps the watermark behind the dropped edits); `changed` is decided by **field comparison** (`editEquals`), not the minute-rounded edit stamp. What Notion can't hold is DECLARED, not approximated: `Integration.capabilities` (base: all true; Notion: no recurrence/reminders/location/cancelledStatus/**timeZone**) drives the editor to HIDE those fields (`getCapabilities` in `Api.ts`, all-true fallback for plain-DTO providers like Dev), and the write mapping throws on recurrence/cancelled as defense in depth. **`description` IS supported: the page BODY ↔ markdown** via `shared/NotionMarkdown.ts` (uses the same `marked` the frontend renders with — now a `src/shared` dependency — so the editor's preview dialect is exactly what Notion receives, incl. `> [!type]` callouts ↔ callout blocks by color). The invariant both directions uphold: *a description edit replaces exactly what the description shows* — `toMarkdown` renders only blocks that are wholly re-authorable (`isReplaceable`, recursive: a bullet hiding an embed is opaque wholesale, as is any block whose children weren't fetched), the write pass (`Notion.replaceBody`) deletes only those (block DELETE = trash, recoverable) and appends the new blocks after the preserved rest, so images (Notion file urls expire hourly), embeds, sub-pages and synced blocks stay invisible AND untouched. The body reader (`fetchBodyBlocks`) descends only into `containerTypes` and exactly `maxNestingDepth`(=2) levels — the same cap Notion puts on write-payload nesting (deeper markdown lists flatten to depth-2 siblings), keeping read and write symmetric. Sync fetches the body for every page the delta serves (a body-only edit within the stamp's minute-rounding must never become permanently skippable, same reasoning as the property apply); a description-only local edit writes blocks then GETs the page for the fresh-stamp echo (no property PATCH), and the stored description is always the write's own normalized markdown (`toMarkdown(toBlocks(input))`) so the next sync compares equal and stays silent. `timeZone: false` because Notion's date property **cannot store a named IANA zone**: the API resolves any `time_zone` you send to a fixed offset and returns `time_zone: null` (verified against a real workspace — a `time_zone` sent with a `Z`/offset start even re-labels the wall clock, corrupting the instant, so `dateFrom` only ever pairs `time_zone` with a naive wall-clock, which preserves the instant). So Notion timed entries always read back `timeZone: null`, no per-entry authoring zone exists, and the zone picker / primary↔entry lens (`EntryDetailsWhen`) is hidden for them — the times still display in the viewer's zone (a view concern, unaffected). Status maps option→group via the data-source schema (page values carry no group); writes send each group's FIRST option id and only DIFFED properties (an unchanged Done never renames a "Shipped" option). A CREATE (and a migration, which routes through `createEntry`) pre-fills the view's own filter properties via `Notion.deriveFilterDefaults(view, dataSource)` — a page is created in the DATA SOURCE, not the view, so without this a task made in a filtered view (e.g. Area=University) never matches the filter and never appears. It reads BOTH `view.filter` AND `view.quick_filters` — real task views keep their filtering in the latter (`filter` is frequently `null`), keyed by RAW property ids while the schema url-encodes them, so `resolve` matches id / url-decoded-id / name. Only conditions a single value satisfies are honoured: select/multi-select/checkbox equality, status equality when the value is a real option (options can't be created on a page write, and the value may be a GROUP name), and **relation `contains <page-id>`** (the actual "Area = University" shape — set the relation to that page). OR branches, `does_not_equal`, formulas, date ranges, and filters on properties that don't resolve are skipped. mitra's own mapped fields always override a filter default (the user's chosen status is never rewritten to satisfy the view). A relation filter only resolves when the RELATED database is also shared with the connection — if it isn't, Notion omits the relation property from the schema/query entirely (it looks deleted), so mitra can't set it; sharing that database is the fix, not code. **The source strictly mirrors the view: there is deliberately NO local retention of a created task that doesn't match** (an earlier `createdHere` pin was removed at the user's request — "all should come from Notion"). A created task appears in the view iff its pre-filled properties satisfy the filter; if the view filters on something no value can set (a pure formula), the task lives in the database but not that view, exactly as in Notion. The deletion pass keeps only two guards, neither of which retains non-Notion content: skip when membership is truncated (10k cap), and a uniform short `localWriteAt` grace window (our own clock) so a brand-new legitimate member isn't deleted-then-re-added while Notion's view index catches up. `applyAndSync` (base) now rejects duplicate `(userId, uri)` connects with a clear error before flush. The add dialog pre-selects only ONE view per database (overlapping views would render a task once per view — a documented, deliberate user choice).
- **Sync pacing lives on the integration**: `Integration.syncInterval` (ms, default 0 = every cycle) lets rate-limited providers poll politely — `GoogleCalendar` and `Notion` return 60s. That pacing IS the quota protection (Google's CalDAV shares the Calendar API limits: 600 req/min/user, 1M req/day/project; an idle sync is ~9 requests, so 60s ≈ 13k/day per account — the DAILY project quota binds first, ~75 connected accounts at 60s vs ~13 at 10s). The `Synchronizer` keeps an in-memory `nextSyncAt` map honoring it, wraps each integration in its own try/catch (one broken account doesn't block the rest of the cycle), and rests a FAILED integration for a **flat** `max(syncInterval, 60s)` — deliberately not exponential: at these volumes extra backoff buys no protection, and flat keeps recovery within a minute of the server coming back. The global 10s cycle stays — it only paces plain CalDAV (self-hosted servers without such quotas) and is what makes remote edits feel live.
- **The integration dialog is two-step on add, and provider-agnostic via polymorphism, not `instanceof`**: the add flow first picks a type from a tile grid, then shows that type's details panel. The tiles are built from the shared registry (`integrationClasses()` — every `@integration`-registered subclass, in registration = display order), NOT a list in the dialog: each connectable class OWNS its own presentation as statics — `label`, `logo` (an asset key), `description` (English source text) — so a provider's identity lives in one place, its class. The entity is constructed ONCE with its final class (`new integrationClass(...)`); there is no in-place type conversion — going back to the picker discards the entity (and any in-flight discovery result, guarded by entity identity). Editing skips the picker entirely: `Integration.editableCopy()` (the mirror of `merge`) hands the form a transient same-class copy — identity + uri kept, `editableCredentials` blanked per provider (CalDAV blanks the password, Google the grant), sources as a plain array (JSON-serializable — a live Collection holds a circular owner reference). The dialog derives its details panel from the `type` discriminator, defaulting to the CalDAV panel for provider types the client doesn't model (e.g. Dev). **Adding a provider = a new `@integration` subclass (with its statics + a `logo`/`canConnect`) + its SVG asset + one panel case + one logo-map entry** — no edits to a central list. NB: the `@model` registrations the API reviver needs survive via `shared/index.ts`'s `export *` because decorator application is a side effect — don't convert those exports to type-only.
- **A provider's identity, presentation, and connect-preconditions live on its class — kept plain so shared/backend stay clean.** These domain classes double as the frontend's API models, so their presentation statics are **plain data only**: `label`/`description` are strings (no `t()` — that global is frontend-only and would break the backend; the dialog localizes `description` at render, and `scripts/i18n.ts` scans the `description` statics so those keys stay tracked), and `logo` is an asset KEY, never a lit template — the SVGs live in `assets/integrations/*.svg`, import as text (esbuild's `.svg` `text` loader, frontend only), and render inline via `unsafeHTML` so the monochrome marks inherit `currentColor` (Google keeps its own gradient). Whether a connection can be attempted is domain knowledge too: `Integration.canConnect` (base: server URL + username; Apple: username only, its URL is fixed; Notion: token only) gates the dialog's Connect button — the dialog adds just the edit policy (an edit always refreshes, the server holds the stored secrets).
- **A fresh `Integration` knows its own `type`**: the `@integration(type)` decorator stamps a `type` static on the class (also its map key) and the base constructor reads `new.target.type` to set the STI discriminator (MikroORM only back-populates `type` on load). ONE `Map<type, class>` (`registeredIntegrations`) serves everything — `integrationClassFor`'s API/backend dispatch, the dialog's `integrationClasses()` tiles, and the self-stamp. So a freshly constructed instance serializes its `type` on the wire and nothing sets it by hand; unregistered internal-only subclasses (Dev) have no `type` static, stay unstamped, and get their discriminator from the ORM on persist.
- **Incoming request bodies are PLAIN DTOs, not rehydrated entities.** `@a11d/api` `structuredClone`s the body before sending, which strips the prototype — so `@type` (a prototype prop set by `shared/model.ts`), `toJSON`, and every **getter** are gone on the wire, and the server's `express.json` reviver never reconstructs request-body models. Backend code reading a request body must touch only **plain data fields** — never call a method/getter on it or rely on `instanceof`. This bit source enabling: `Integration.applyAndSync` keyed `incoming.sources` via the `Source.key` **getter** (→ `undefined` on a plain object → every source disabled). Fixed with the static `Source.keyOf({type, uri})` (the getter delegates to it) for the incoming side. Key/compare incoming-body fields by hand, not via a getter.
- **All-day:** `Entry.allDay` is a **persisted flag**, not inferred from midnight times. CalDAV reads it from a date-only `DTSTART` (`event.startDate.isDate`) on sync and writes `DTSTART;VALUE=DATE`/`DTEND;VALUE=DATE` (exclusive next day) on create/update; `start`/`end` still carry the day bounds (midnight → exclusive next midnight) for rendering. Always set the flag where an entry is created (drag, seed, sync).
- **All-day bounds are calendar DATES, not instants** (`shared/calendarDate.ts`): stored canonically as **UTC midnights**, which is zone-less and deterministic — the deployment container's TZ is irrelevant. They cross the API in the VIEWER's zone: every entry read/write sends `?tz=<browser IANA zone>`; reads project the dates to the viewer's local midnights (`projectAllDay`, entries.ts `projectedForViewer`), writes normalize the viewer's midnights back to dates (`normalizeAllDay` — including scoped-edit/delete `recurrenceId`s, which round-trip through projected data). So an all-day event covers the same dates, midnight-to-midnight, in every browser zone. Rules that follow from this: NEVER `toJSDate()` a date-only ical.js value (server-local midnight — the prod overlap bug; use `CalDAV.instantFrom`, which reads y/m/d → UTC); all-day series expand/shift/exclude in **UTC** (`occurrences.ts dayMathZoneOf` — `entry.timeZone` governs only TIMED wall-clock math). **No data migration for pre-convention rows** (matching the OIDC precedent): rows written before this convention keep their old zone-shifted midnights — users **re-import** their integrations to re-canonicalise, so don't write backfill code. Known trade-off: all-day reminders anchor to UTC midnight (per-viewer reminder zones are a future concern).
- **Background Sync Architecture:** All data synchronization should occur entirely in a background daemon (e.g., `server.ts` loop) that polymorphically iterates over all base `Integration` records. Do NOT block frontend API reads with live external syncs; the frontend `/api` routes must always query instantly from the local SQLite store.
- **CalDAV Sync Quirks:** When syncing CalDAV via `tsdav`, **do not** use `smartCollectionSync` or `smartCollectionSyncDetailed`. It has a hardcoded bug that filters out any calendar URLs that do not end in `.ics`, breaking platforms like some CalDAV servers. Always implement manual WebDAV XML syncing by passing `client.syncCollection()` directly and fetching `changedUrls` manually.
- **Multiget must tolerate per-member 404s.** `client.fetchCalendarObjects({ objectUrls })` runs ONE `calendar-multiget` REPORT, and tsdav's `collectionQuery` **throws if ANY member response is ≥ 400**. Google (and any server) can report a member that vanished between the sync-collection listing and the fetch with a per-href 404 — on an incremental delta especially — which would otherwise abort the entire source sync. Worse, the abort happens before the caller flushes the advanced sync token, so the *same* delta is retried forever (a `resyncSource` that wipes the token dodges it — hence "reimport fixes it, save doesn't"). `CalDAV.fetchObjects` wraps the multiget: batch first (fast path, one request), and on throw refetch each object individually, dropping the gone ones so the sync completes and the token advances. Route the changed-object fetch through it, never `fetchCalendarObjects` directly.
- **Date writes preserve the resource's AUTHORED FORM** (`CalDAV.writeDate`): a `DTSTART`/`DTEND`/`DUE`/`EXDATE`/`RECURRENCE-ID` that carried a `TZID` is rewritten as that zone's WALL CLOCK (via the resource's own VTIMEZONE), keeping the parameter; zoneless values stay UTC (`toICALTime`), with any stale `TZID` parameter dropped. Never `updatePropertyWithValue(name, ICAL.Time.fromJSDate(date, true))` directly on an existing date property — ical.js keeps the old TZID parameter, the zone reinterprets the UTC wall clock, and the event shifts by the zone offset on zoned servers (Google authors TZID forms; a Berlin series moved to 10:00 landed at 08:00 local this way, and Google then detached its orphaned exception into a standalone event). A fresh `EXDATE`/`DTEND` follows DTSTART's form (RFC 5545 matches instances by it). Also: **a series-wide start shift shifts bundled override `RECURRENCE-ID`s** (components + local rows) by the same delta in `updateEntry` — an unshifted override orphans, and the expansion renders both the shifted occurrence and the exception. The overrides' own DTSTARTs stay (a custom-timed exception keeps its custom time).
- **Google bumps the etag AGAIN after accepting a write** (async normalization), so a second edit of the same resource within a sync cycle carries a stale If-Match and would 412 spuriously. Every CalDAV write therefore goes through **`CalDAV.writeResource(entry, applyTo)`**: the edit is a pure `raw → raw` transformation (`applyTo`), attempted with the stored etag and — on a 412 — re-applied ONCE onto the refetched current resource with its fresh etag (`refetchResource`). This keeps the guard's meaning: a genuine concurrent change simply becomes the base the field edit merges onto (the same semantics any edit has), and a second 412 propagates. `deleteEntry` retries its DELETE the same way (and treats a 404 as success — already gone). When adding a write path, express it as an `applyTo` closure (idempotent — it may run twice; anything non-idempotent like row mutations or delta computations stays OUTSIDE) and route it through `writeResource`, never `updateCalendarObject` directly.
- **Opt-in Data Flow:** Newly discovered external sources (like new calendars) should always be persisted to the DB with `enabled: false`. Do not perform heavy `Entry` syncing loops until the user explicitly enables a source.
- **Source reconciliation preserves LOCAL renames** (`Integration.getSources`): a user can rename a source (`PUT /sources/:id/name`), and a background sync must not reset it to the provider's name — the previous bug (`match.name = source.name` unconditionally, every cycle). `Source.remoteName` (nullable column, auto-added by `orm.schema.update()` on boot — no manual migration) stores the provider's name as of the last reconcile. Reconcile rule: a **new** source records `remoteName = name`; a **matched** source with a `null` baseline (a row predating the column) records the provider name silently WITHOUT touching the displayed `name` (so an existing rename survives the first post-upgrade reconcile); otherwise the provider name is adopted only when it **changed** since last reconcile (a genuine REMOTE rename — a local custom name yields to it), and an unchanged provider name leaves the row's current name alone. Same pattern would apply to any per-source field a user can override locally (only `name` today; `color` is already write-once-then-user-owned).
- **Incoming request bodies are PLAIN DTOs, not rehydrated entities.** `@a11d/api` `structuredClone`s the body before sending, which strips the prototype — so `@type` (set by `shared/model.ts`), `toJSON`, and every **getter** are gone on the wire, and the server's `express.json` reviver never reconstructs request-body models. Backend code reading a request body must touch only **plain data fields** — never call a method/getter on it or rely on `instanceof`. This bit source enabling: `Integration.applyAndSync` keyed `incoming.sources` via the `Source.key` **getter** (→ `undefined` on a plain object → every source disabled on save). Fixed with the static `Source.keyOf({type, uri})` (the getter delegates to it) for the incoming side. Key/compare incoming-body fields by hand, not via a getter.

## Multi-user & authentication (OIDC)

- **Zero-auth single-user is the default**: `authMiddleware` attaches the seeded `[default_local_user]` to every request. Setting `MITRA_OIDC_ISSUER` (+ `MITRA_OIDC_CLIENT_ID`, `MITRA_URL`, optional `MITRA_OIDC_CLIENT_SECRET`/`MITRA_OIDC_SCOPES`) switches on multi-user mode; a half-configured issuer fails the boot loudly (`Oidc.fromEnv`).
- **The BACKEND is the OIDC relying party** (`backend/Oidc.ts`, Authorization Code + PKCE via `openid-client`, routes under `/auth/*`) — deliberately NOT frontend `oidc-client-ts` + Bearer tokens: the SPA is same-origin (one Express serves both), `@a11d/api` already sends cookies, and tokens staying server-side beats XSS-exposed browser storage. Discovery is lazy and retried (the IdP may boot after mitra in the same compose stack); an `http:` issuer is allowed for LAN/compose-internal IdPs.
- **Sessions are mitra's own** (`backend/Session.ts`): an opaque 256-bit cookie token stored SHA-256-hashed as the row id, sliding 30-day expiry. The IdP's tokens are discarded after sign-in — mitra calls no upstream APIs for the user — except the raw `id_token`, kept per session as the `id_token_hint` for RP-initiated logout. CSRF rests on `SameSite=Lax`. Unauthenticated requests split by kind: `/api/*` → 401 (the frontend's `ApiError` bounces through `/auth/login`), asset paths (with an extension) pass — bundles and the PWA manifest are public code — and page navigations redirect to `/auth/login?returnTo=…`.
- **The OIDC identity is a value object, not flat columns**: `Identity` (`shared/Identity.ts`) holds `issuer`/`subject` (the key — a subject is unique only within its issuer) plus the vouched `email`/`name`/`picture` (the photo is stored as a LINK only — the browser loads it from the provider), embedded into `User` as `oidc_*` columns via `@embedded(() => Identity, { prefix: 'oidc_', nullable: true })` — the same MikroORM embeddable pattern as `Recurrence` on `Entry` (registered in `orm.ts` entities; unique on `['identity.issuer', 'identity.subject']`). Absent entirely in single-user mode, so `user.identity` presence IS the "is this an OIDC user" test (frontend + backend). Behavior lives on it: `Identity.fromClaims` / `applyClaims`.
- **JIT provisioning is domain behavior**: `User.provision(em, issuer, claims)` matches on the embedded `(issuer, subject)` and creates a fresh user on first sight. `email`/`name` refresh from the ID token on every sign-in. **No data migration**: enabling OIDC does NOT carry the single-user default user's data over — every identity (incl. the first) starts empty and re-adds its own integrations. This is a deliberate call (the user's): don't write upgrade/backfill code; tell users to reimport.
- **Ownership scoping lives on `User`**: routes resolve entities exclusively through `user.integrations/integration/sources/source/entry` — a foreign id throws `NotFoundError` → the central 404, never leaking another user's data. Never use a bare `em.find*` for user-owned entities in a route unless ownership was already proven upstream in that handler.
- **Everything user-adjacent is per-user**: `syncEmitter.emit('updated', userId)` (the SSE feed filters by `req.user.id`; `Synchronizer` collects changed owners per integration), push subscriptions carry a required `userId` set at subscribe time (re-registration on boot reassigns a shared browser to whoever is signed in) and reminders route via entry → source → integration → owner (`sendTo(userId, …)`).

## Calendar layout & segmentation (current model)

- **View components are named after the unit they strip**: `Days` (`mitra-days`) is the week view's horizontal strip of day columns; `Weeks` (`mitra-weeks`) is the month view's vertical strip of week rows. A future view follows the same scheme (e.g. a year view = a strip of months). The user-facing *view* names ("Week"/"Month") and the drag controller's `grid` discriminants stay as-is.
- **`EntrySegments` (frontend cohort) is the whole layout engine** — small, because CSS does the placing. Build via `EntrySegments.of(entries, days)` (memoised on `days`). API:
  - `EntrySegments.for(entry)` — the entry's per-day segments, linked `previous`/`next`, memoised so instances are stable across renders.
  - `timedOn(day)` — the day's timed segments clustered into side-by-side columns (the *one* genuinely cross-segment computation; writes each segment's `overlap = {slot,total,span}`).
  - `runsIn(from, to, accept)` — one representative segment per matching entry whose run touches the window, sorted so DOM (and paint) order stays deterministic. Used for the week all-day lane and per-week in month.
  - `monthSlots` / `allDaySlots` / `monthWeek(week, maxSlots)` — computed lane packing (one shared greedy `slots` pass). Month needs exact lanes for its "+N more" overflow; the week's all-day lane needs them for its explicit row tracks (`allDaySlots` packs only all-day entries — timed ones never hold a lane there).
  - `static laneRank(entry)` — lane ordering for month packing (a view concern, hence here and not on `Entry`).
- **Grid gestures — one `EntryDragController`, both views, three gestures.** Attached to the grid *container* (`Days`/`Weeks`), never per-`Day`, so a gesture can span day cells; the view selects the surface with a `grid: 'week' | 'month'` arg. It snapshots cell geometry once at pointer-down (so each move is pure arithmetic — no per-frame reflow), coalesces moves to one update per animation frame, captures the pointer on the host, and tears down on pointerup **or pointercancel**. Pointer-down classifies the gesture once (a `kind` discriminant): a `.resize-start`/`.resize-end` handle → **resize**, a persisted entry's body → **move**, empty grid/lane/cell → **create**. It also **owns tap-to-open** (a no-move tap on a body opens the editor) — `mitra-entry-segment` has no click handler, so a drag can't fire a stray open on the wrong slice. Geometry/granularity are shared: timed only in the week's grid (minutes off the 1440-row grid, snapped to `snapMinutes`); the all-day lane and the *whole* month are day-granular (a timed entry moved in the month shifts whole days, keeping its clock time).
  - **create** — week timed/all-day lane/month, as before (plain click: nothing in week, quick-create a day in month). New entries target the dev sample calendar in development.
  - **move** — translate a persisted entry preserving duration (clone both ends by one delta; the result's leading edge snaps to grid). Allowed for every entry, including timed-in-month.
  - **resize** — drag one edge while the other stays fixed, reusing the create min/max-flip placement (`entryPlacement.ts`), so dragging an edge past the other flips gracefully and multi-day spans fall out for free. **Pure placement math lives in `entryPlacement.ts`** (`placeTimed`/`placeAllDay`/`resizePlacement`/`snapToGrid`) — dependency-free and unit-tested (the all-day exclusive-next-midnight bound makes the fixed *trailing* edge `end − 1 day`).
  - **Resize handles** are 0.25rem strips rendered by `mitra-entry-segment` per a `resize: 'block' | 'inline'` prop the view sets (Day timed → block/vertical, all-day lane + month all-day bars → inline/horizontal, **month timed bars → none**, gated on the entry's *persisted* `allDay` flag, not the segment's derived one). CSS hides a handle at a clipped/interior edge via the same `has-previous`/`has-next` attributes (so `MonthBar` exposes `clippedRight`) and drops block handles on too-short entries.
- **Drafts & optimism — one pending local entry, new *or* edited.** A locally-**created** entry (drag/click) has **no id** (the backend assigns one on create, so `Entry.persisted` = `id != null` *is* the draft test — no flag, no status enum). An **edit** (move/resize) is a *clone* of an existing entry with mutated start/end (same id) — never an in-place mutation, so a cancelled gesture leaves the original untouched. `DraftController` (static store) holds the one pending entry: `mergeInto` substitutes it over the server copy by id (an edit) or appends it (a not-yet-saved create); `reconcile` (called from the page's fetch) drops it once the server echoes the **matching content** (start/end/allDay) — id alone would drop an edit's overlay before its update lands and snap it back. **Commit:** create → `createEvent` (no id), set the returned id in place; move/resize → `updateEvent(clone)`, reverting *that* draft on failure. A `dragging` flag floats the actively-dragged entry above its cluster. Components attach an instance only to **subscribe**.
- **Backend owns ids:** `Entry.id` has no default; assign one at every persist site (the POST route — which covers both providers' `createEntry` — plus `seedDev` and CalDAV sync). The frontend never mints ids; a missing id means "not yet saved".
- **Hot-loop date math:** `@3mo/date-time` rebuilds a `Temporal.Instant` **and** `ZonedDateTime` on *every* accessor (`.hour`/`.dayStart`/`.equals`/`.isBefore`…), which dominates per-frame render/clustering loops. So compare **cached epoch-ms day numbers** instead: `EntrySegment` memoises its numeric projections (`startMinute`/`endMinute`/`dayValue`) and segments are globally memoised (`EntrySegments.for`), so the Temporal cost is paid once per segment; compare `dayValue`/`fallsOn(dayValue)` (or hoist a `dayStart.valueOf()` out of the loop) rather than `dayStart.equals(...)`. The richer `DateTime` comparison API still applies to non-hot code.
- **Self-placement:** views map a date → grid column themselves (build a `Map<dayValue, index>` once per render — never `findIndex` inside `.map()`) and set `grid-column`/`grid-row`/`--overlap-*` inline. The all-day lane self-places columns by date and rows from `allDaySlots`. `CalendarLayout` (an old shared class) was removed — don't reintroduce it.
- **The week all-day lane has EXPLICIT row tracks, never auto-flow rows behind a `min-block-size`.** The view computes the window's lane count (`allDaySlots`, +1 trailing empty lane as the drag-to-create target and the move ghost's preview row) into `--_all-day-rows`, and renders a per-day `.day` surface cell spanning `grid-row: 1 / -1` so the day columns' colour and 1px separators continue through the strip. Why not `grid-auto-flow: row dense` + `min-block-size` (the old model): the timed row is measured to fill the viewport exactly, so `mitra-days` never has free space and the lane's `auto` row sits at its *base* size — and a specified min-size REPLACES the content-derived minimum there, freezing the lane at the min while extra bars overflow onto the timed grid. Beyond `Days.MAX_VISIBLE_LANES` (4) the strip stops growing and scrolls within itself (`[data-all-day-overflow]`, attribute-gated because a scroll container would capture the bar titles' inline stickiness even when nothing overflows).

## Yearly view (the months strip)

- **`mitra-months` (frontend/Months.ts) is the year view — and it's a strip, not a "year"**: the vertical sibling of `Days`. One row per month scrolling seamlessly across year boundaries on the same `CalendarDatesController` 3-year buffer (`radiusDays: 240`), one column per weekday-aligned day slot (`31 + daysInWeek - 1` = 37), so weekdays line up vertically across the whole year like a paper wall planner. "Year" is just this strip at its default density (12 rows fill the viewport); custom spans (quarters, "X months") are density presets, not new components.
- **Cells are `mitra-day`, not bespoke divs** — the day component's own container queries carry it down to year scale: a new compact tier (`@container ((max-height: 8rem) or (max-width: 3rem))`, unreachable by the month's ≥8.5rem cells and the week's ≥10rem columns) collapses the header to a bare centred numeral and hides the weekday/month parts. Extend `mitra-day`'s versatility for a new scale; never fork a parallel cell.
- **Month grouping lives on `CalendarDatesController`** (`get months(): CalendarMonth[]`) — deriving calendar structure from the buffer is the controller's job, not a view's. `CalendarMonth` is a thin **value object** over the month's consecutive days (`number`/`firstColumn`/`columnOf`/`intersects`), computed on read with **no cache**: `@3mo/date-time`'s `DateTime` memoises every accessor per instance (`zonedDateTime`/`dayStart`/`month`/…) and the buffer's day instances are stable across renders, so the grouping is cheap plain-array work and precomputing/caching day-start keys buys nothing. Buffer days carry the buffer's *generation clock time*, so `firstValue`/`lastValue` and `data-date` still go through `.dayStart`.
- **Lanes are pure CSS, overflow clips**: bars render into a per-month `.entries` overlay (subgrid columns, `grid-auto-rows: 1rem`, `grid-auto-flow: row dense`, `pointer-events: none` with interactive bars) — like the week's all-day lane, no JS lanes and no "+N more". DOM order is the packing priority: a stable sort by `EntrySegments.laneRank` (multi-day arcs → all-day → timed) so the entries that shape a year keep the visible lanes; a too-dense row clips under a bottom fade mask, and zooming in is the reveal. The segments' own container queries shed time/labels as lanes tighten — that's the whole density-adaptation story.
- **Density zoom = one abstract `DensityController`** (wheel/pinch/ease/persist mechanics) with per-view subclasses owning what zoom *means*: `DayDensityController` (`--_week-zoom` multiplier — the timed row is the pure CSS formula `zoom × (100% − header row − all-day row − gaps)`, day-fraction pinning measured once per gesture) and `MonthsDensityController` (`--months-zoom` multiplier, CSS-derived row height `(100% - header) / 12 * zoom`, ratio-scaled `scrollTop` pinning). **Zoom NEVER sizes tracks via measured px**: every `mitra-days` row is definite, because the row-spanning `mitra-day` subgrids give intrinsic rows unbounded growth limits — any free space (e.g. a grown window meeting a stale px row) pours into them and the subsequent remeasure reads its own bloat back, locking the layout (the historic resize bug). The base registers its listeners MANUALLY — `@a11d/lit`'s `@eventListener` keeps initializers in a set subclasses share with their base (broken copy-on-write), so a decorated subclass would leak listeners onto every sibling. Hosts whose scroll handler derives state from the scroll position must skip frames while `density.active` — per-frame pinning otherwise walks the navigating date (and its ±9-month fetch window) through every intermediate month; the controller re-dispatches one `scroll` on settle.
- **The strip owns its scroll position — the browser must not**: `overflow-anchor: none` on the host (native scroll anchoring fights the windowed rows swapping above the viewport and can runaway-feed the scroll→date handler), and when a scroll-driven `navigatingDate` set regenerates the buffer (`buffer.days` identity changed), re-anchor immediately via `scrollToDate` — every row just shifted under the unchanged scroll offset.
- **Gestures**: `EntryDragController` grid `'year'` — day-granular all-day create/move/resize like month (click quick-creates); every view (week/month/year) renders its cells as `mitra-day`, so the controller matches/snapshots them by that tag (each carries `data-date`, which the snapshot reads). The month rail navigates to the month's **middle** and fires `switchToMonth` — the month view centers the target's *week*, and day 1 often sits in a week still belonging to the previous month.
- **A new Light-DOM component's `static styles` MUST be registered in `Mitra.ts`'s aggregated styles** — light-DOM components (`createRenderRoot() { return this }`) get no automatic style adoption from lit; the app shell collects every component's styles into the one global sheet. Forgetting this renders the component completely unstyled with no error.

## Command palette

- **`mitra-command-palette` is a pure view; commands belong to their context.** The `Command` interface (`frontend/Command.ts`: heading, icon, optional shortcut hint + match keywords, `execute`) is defined by the page that owns the behavior — `PageCalendar` builds its command list per render (so view-dependent labels like "Next Week/Month" stay current) and passes it in as a property. The palette only filters (`commandMatches`: every query term must appear in heading+keywords), navigates (arrows/Enter, hover) and dispatches. Picking an entry emits `navigate` with its start **and** calls `EntryStore.requestOpen(id)` so its editor pops open once its segment renders; picking a command runs `execute()` after closing.
- **Opening a picked entry reuses the draft auto-open mechanism, generalized.** `EntryStore.requestOpen(id)` sets a pending `openEntryId` (sibling to the draft's `autoOpen` flag); `EventSegment.updated` opens the run-start segment (`!hasPrevious`, so a multi-day entry opens once) when `shouldOpen(entry)` matches, then `consumeOpen()`s it. The id is kept until consumed so it survives the async refetch the navigation triggers (cross-month picks work), and `shouldOpen` matches `entry.recurrenceMasterId === openEntryId` too, so picking a recurring master opens whichever occurrence the navigation lands on.
- **Entry search is backend-side and unwindowed** (`GET /entries/search?q=`): LIKE over heading/description/location across visible sources (limit 20, newest first), because the calendar only ever fetches ±1 month — the palette must find *everything*. Recurring masters match as themselves (one row stands in for its series). Debounced 200ms with a monotonic token so only the latest response lands.
- **Open affordances:** Ctrl/Cmd+P (window keydown listener owned by the palette itself; `CommandPalette.hotkey` renders the platform label) and the header's fake search box — a `button.search` styled like an input, centered between flex spacers, collapsing to an icon-only button under `@container (max-width: 40rem)` (the header is an inline-size container).
- **It's a native `<dialog>` with `closedby="any"`** for light dismiss — no hand-rolled backdrop-click handler (that pattern closes on the *trigger's* double-click). Styling follows the popover look: translucent `color-mix` surface + `backdrop-filter: blur`, `@starting-style` entry animation.
- **"Create Entry" reuses the gesture-draft flow** (`EntryStore.upsertDraft` + `openDraft`): a blank one-hour draft at the next full hour today on `getPrimarySource()` (Api.ts — default-or-first-visible; also what the drag controller uses). It always `setView('week')`s first: the week's timed grid renders every draft, whereas a crowded month cell folds it into "+N more" and the auto-open editor would never appear.
- **Single-letter view shortcuts** (`PageCalendar.handleKeyDown`: `w`/`m`/`y`/`t`, a window keydown listener) MUST stand down for any editable target — `<input>`, `<textarea>`, `<select>`, AND `contenteditable` (the sidebar's inline source-rename is a contenteditable div, so typing "m" once switched to Month mid-rename), plus Ctrl/Cmd/Alt chords and `e.isComposing` (IME). Guard on `isContentEditable`, not just the element types. Any future bare-letter shortcut needs the same guard.

## Recurrence (RFC 5545)

- **One master row, occurrences expanded on read — never stored.** A recurring series is a single `Entry` carrying `rrule` (+ `uid`); `CalDAV.expandRecurrence(raw, windowStart, windowEnd)` iterates the rule into the occurrences intersecting the visible window. `GET /entries` **excludes masters** (`rrule != null`) from the plain rows and emits synthetic per-occurrence `Entry` objects instead, so the bare master (whose `DTSTART` is only its first instance) is never rendered on its own. Occurrences are non-persisted, with a **CSS-ident-safe id** `` `${master.id}__${epochMs}` `` (underscores + digits — colons/dots break `anchor-name`/`view-transition-name`) and `recurrenceMasterId` set.
- **Overrides & exclusions.** A single edited occurrence syncs from CalDAV as its own row with a `RECURRENCE-ID`, linked to its master by shared `uid` (a post-loop pass in `syncSourceEntries`, scoped to the source); the GET skips expanded dates that already have an override row. `EXDATE`s are applied during expansion (matched by epoch-ms).
- **One resource, MANY rows.** A compliant server (Google, iCloud) bundles a series master and ALL its override components into ONE `.ics` resource (RFC 4791: one UID per resource) — never assume one component per resource. `syncSourceEntries` therefore upserts one row per `vevent`/`vtodo` component; within a resource a row's identity is its `RECURRENCE-ID` (the master has none) — the unique index is `(source_id, uri, recurrence_id)`, NOT `(source_id, uri)`. Consequences for the write paths: select the row's component via `CalDAV.componentFor` (never `getFirstSubcomponent` — that edits whichever component happens to be first); after any resource write, `syncResourceRows` mirrors the new `raw`/`etag` onto the resource's other rows (each keeps a copy — a stale sibling etag would 412); `excludeOccurrence` also strips a bundled override component of that instant (EXDATE alone only prunes the recurrence SET — the override would keep the instance alive) and its local row; `deleteEntry` on an override row routes through the master's exclusion — deleting the shared resource would kill the whole series. An override that vanishes from the resource (occurrence reverted) loses its row on the next ingest. Rows synced BEFORE a parsing improvement don't self-heal (unchanged resources aren't re-listed by sync-token deltas) — "Re-import entries" is the recovery path.
- **Editing is series-wide (current version).** There is no per-occurrence editing yet. Editing a rendered occurrence routes to the master via `recurrenceMasterId` (`Api.updateEvent` sends only series-wide fields, so the master keeps its own `start`/`end`); deleting removes the whole series. **`Entry.partOfSeries`** (`!!rrule || isRecurring`) is the intrinsic test for "belongs to a series" — its schedule (date/time/all-day) is read-only and it isn't drag/resize-movable; only the Repeat field stays editable. Setting a repeat on a non-recurring entry makes it a master (its own id; the schedule then locks — remove the repeat to edit times again).
- **RRULE strings are authored on the frontend.** `src/frontend/recurrence.ts` owns the whole RRULE subset: `buildRRule`/`parseRRule`, the date-derived quick presets (`recurrencePresets`), structural equality/match, and the human `recurrenceLabel`. `mitra-repeat-field` (the Notion-style trigger + preset dropdown + Custom dialog) mutates `entry.rrule` in place and fires a distinct `recurrenceChange`. The **backend never builds rules** — it just writes the string onto the master's `.ics`: `CalDAV.createEntry`/`updateEntry` set it via `ICAL.Recur.fromString(rrule)` and drop it (collapsing the series) via `removeAllProperties('rrule'|'exdate')`. The routes validate it first (`CalDAV.isValidRRule`, 400 on junk) so a bad rule can't 500 the `.ics` writer.
- **Recurrence edits go through a dedicated path, never the content-edit path.** `Api.updateEvent` (heading/description/colour/status, plus the move-to-master routing for occurrences) **never writes `rrule`**; only `Api.setRecurrence` does (string = set, `null` = remove; `entries.ts` PUT maps absent = keep). This split is load-bearing: a synced *override* (an externally edited single occurrence — `recurrenceMasterId` set, but no `rrule` of its own) renders as an editable row, and inferring `rrule` from it on a benign rename would otherwise wipe the master's whole series. So content edits must stay rule-silent.
- **Expansion works without an `.ics` too.** `expandRecurrence(raw,…)` (CalDAV, carries EXDATEs/anchor from the stored `.ics`) and `expandRecurrenceFields(rrule, start, end, …)` (column-only, for integrations like `Dev` that persist no `.ics`) share one `iterateOccurrences` core; `entries.ts` picks the column path when `master.data.raw` is absent. **ical.js 2.x has no `Recur.between()`** — the core iterates `rrule.iterator(anchor)` and stops once occurrences ascend past the window, with a generous iteration cap (a far-future window must still be *reachable* for a dense daily series) and a no-advance guard against a malformed rule.
- **Backfill:** entries synced before recurrence support have the `RRULE`/`RECURRENCE-ID` in `data.raw` but NULL columns, and the etag-skip won't re-read them — `backend/orm.ts` parses the candidates once on boot and links overrides to masters by UID (idempotent).

## Build / test / run

- **Node 25+** is required at build & runtime (Temporal API behaviour) — pinned in the Dockerfile and CI.
- **Typecheck (one-shot):** run `tsgo` (`node_modules/@typescript/native-preview-<platform>/lib/tsgo[.exe] --noEmit`). esbuild does NOT typecheck.
- **Lint:** `npm run lint` → `eslint .` (flat config `eslint.config.mjs` extending `@a11d/eslint-config`). Pinned to ESLint 9 (`eslint-plugin-lit@1` breaks on ESLint 10). Style is enforced: tabs, single quotes, **no semicolons**, **no trailing newline** (`eol-last: never`), `no-console` allows only `warn`/`error` (off in `scripts/**`), `^_`-prefixed args are ignored. Run `eslint . --fix` after edits.
- **Tests:** `npm test` → `scripts/test.ts` esbuild-bundles `src/**/*.test.ts` → `out_test/`, then `node --test`. Use the Node built-in runner (`node:test` + `node:assert/strict`) with real `DateTime`.
- **Dev:** `npm start` → `scripts/dev.ts`: `tsgo --watch` + esbuild watch for backend (`out/server/server.mjs`, run via `node --watch`) and frontend (`dist/`).
- **Production build:** `npm run build` → `scripts/build.ts` (one-shot, no watch): backend bundle + minified frontend → `dist/`. The esbuild config (entry points, externals `better-sqlite3`/`tsdav`, the `createRequire` banner, the `index.html` shell) is shared between dev and prod in `scripts/esbuild.ts` — change it there, not in two places. `data/` must exist before boot (better-sqlite3 won't create it); a committed `data/.gitignore` keeps the dir, and the Dockerfile `mkdir`s it.
- **Dev sample data:** `src/backend/Dev.ts` `seedDev(orm)` **persists** a real `Dev` sample integration + sources + entries (idempotent, keyed by a stable id; never deletes), gated by `process.env.MITRA_DEV` (set only by `scripts/dev.ts`). It seeds real rows — *not* a read-only overlay — so hiding sources, editing, creating and deleting all work through the normal routes. The user can simply delete the integration when done. Seeding runs once; a `npm start` restart re-checks but won't duplicate.

## Docker, CI & releases

- **Image:** multi-stage `Dockerfile` (build → `npm prune --omit=dev` → `node:25-bookworm-slim` runtime). Both stages share the base so the prebuilt/compiled `better-sqlite3` binary is ABI-compatible (the builder has `python3`/`make`/`g++` so it compiles if no prebuild exists for the Node 25 ABI). The `org.opencontainers.image.source` LABEL links the image to the repo on GHCR. Published to `ghcr.io/a11delavar/mitra`.
- **Tagging is `docker/metadata-action`, not hand-rolled** (`.github/workflows/docker.yml`): push to `main` → `dev` + `sha-<short>`; **push tag `vX.Y.Z`** → `X.Y.Z`/`X.Y`/`X`/`latest`; pre-release tag `vX.Y.Z-rc.N` (channels `rc`/`beta`/`alpha`) → only the precise tag (`flavor: latest=auto` auto-skips floating tags + `latest` for pre-releases); PRs build-only. Multi-arch amd64/arm64 via QEMU+buildx. Image tags come off the **git ref**, so they don't depend on a GitHub Release existing.
- **Changelog lives in the repo** as `CHANGELOG.md`, generated from commits by **git-cliff** (`npm run changelog`, config `cliff.toml`) and then hand/AI-editable. Release flow: `npm run changelog -- --tag vX.Y.Z` → edit the new section → `commit -m "release: vX.Y.Z"` → `git tag vX.Y.Z && git push origin main vX.Y.Z`. **Order matters:** the changelog edit must be committed *before* the tag points at it (the release body is extracted from the file at the tagged commit).
- **Releasing = pushing the SemVer tag.** That one push fans out (in parallel) to `docker.yml` (builds images off the git ref) and `release.yml` (publishes the top section of `CHANGELOG.md` as the GitHub Release body; pre-release if the tag has `-rc`/`-beta`/`-alpha`). Tag-driven on purpose — avoids the `GITHUB_TOKEN`-created-release-doesn't-trigger-workflows gotcha. The README "Releasing & versioning" section is the user-facing source of truth.
- **GHCR has no GC** → `.github/workflows/cleanup.yml` prunes untagged manifests + old `sha-*` tags (keeps `latest`/`dev`/releases). Ships in `dry-run` — verify before enabling.
- **Quality Assurance** (`.github/workflows/qa.yml`): three **parallel** jobs — type-check (`tsgo`), lint (`eslint`), test (`npm test`) — on push/PR. Publishing untested images is not acceptable.

## Conventions

- **Commits:** single-line `type: Capitalized phrase`. The type is load-bearing — `CHANGELOG.md` is generated from commit subjects (git-cliff, `cliff.toml`), so it decides the changelog section: `feat:`→Features, `fix:`→Bug Fixes, `perf:`→Performance, `refactor:`→Refactors, `docs:`→Documentation, `test:`→Tests, **`ci:`/`build:`/`infra:`→Infrastructure**, `chore:`→Chores, anything else→Other. `release:` commits are excluded. Commit ONLY when explicitly asked.
- **The commit subject IS the release note** — it lands verbatim in `CHANGELOG.md`, so write it for end users, not for the diff: state the visible change or benefit in plain language and avoid implementation detail (component/file names, and internal mechanics/jargon like *popover*, *pointer capture*, *light-dismiss*). Prefer the effect the user sees ("Keep the entry editor opening reliably when clicking an entry") over the cause ("Defer showPopover past the captured-pointer click"). `refactor:`/`chore:`/`ci:` subjects are the exception — they have no user-facing effect, so describe the internal change plainly.
- Indent with **tabs**.
- **License:** the project is **AGPL-3.0-only** (`LICENSE`, `package.json`, image label). Keep new workspace manifests consistent.
- Public repo: keep secrets/credentials and personal identifiers out of committed files; leave the git author config unchanged.
